# 設計: CLI ランタイム共通

## 1. 目的

- [仕様書](../spec.md) の 7.2、7.3、7.4、7.8、9 を詳細化する。
- CLI コマンド共通の引数処理、依存解決、子プロセス起動契約を整理する。

## 2. 設計対象

- CLI エントリポイント
- npm パッケージの `bin` 契約
- コマンドディスパッチ
- `dotenvx` 実行ファイルの解決
- 子プロセス起動と終了コード伝播
- 人間向けメッセージ出力

対象外:

- `.dotenvx-keychain` の内容決定
- シークレットストアの OS 別実装
- `dotenvx` の暗号化仕様

## 3. エントリポイントと引数処理

### 3.1 公開名と `bin` マッピング

- 公開 npm パッケージは `package.json` の `bin` で
  `dotenvx-keychain` と `dxk` の 2 つの実行名を公開する。
- 2 つの実行名は同一の JavaScript エントリポイントへ割り当てる。
- `npx dotenvx-keychain ...` とローカル導入後の
  `dotenvx-keychain` / `dxk` は、
  同じディスパッチ実装へ到達しなければならない。
- 公開 tarball からの実行で完結するよう、
  エントリポイントはリポジトリ相対パスへ依存しない。

### 3.2 想定コマンド一覧

- `init [id]`
- `run -- <command> [args...]`
- `list` (`ls`)
- `remove <id>` (`rm`)

エントリポイントは `dotenvx-keychain` と `dxk` の 2 つの
バイナリ名を同一実装へ割り当てる。
最初の位置引数はコマンド名として解釈し、
`ls` と `rm` はそれぞれ `list` と `remove` へ正規化する。
`init` と `run` は正規名のみを受け付ける。
短縮形は正規名と同じ引数契約、メッセージ、終了コードを持つ。

### 3.3 使用法エラー

- 未知コマンド
- `run` の子コマンド未指定
- `remove` の ID 未指定

使用法エラーは次の共通規則を持つ。

- 終了コードは `2` とする。
- エラーメッセージは標準エラーへ出す。
- 1 行目でエラー理由を示し、2 行目以降で使用法を示してよい。

### 3.4 コマンドディスパッチ

ディスパッチの責務は次に限定する。

1. 引数配列からコマンドを識別する。
2. 共通依存を初期化する。
3. 各コマンドハンドラへ引数を渡す。
4. 返された終了コードをそのまま `process.exitCode` へ設定する。

ハンドラは少なくとも次の形を想定する。

```ts
type CommandHandler = (args: string[]) => Promise<number>;
```

### 3.5 実装基盤

- 実装言語は TypeScript とする。
- モジュール形式は Node.js ESM とする。
- `package.json` の `engines.node` は `>=20` を下限にする。
- ビルドは `tsc` による `dist/` 出力だけで開始し、
  bundler は v1 初期実装では導入しない。
- npm scripts には少なくとも `build`、`typecheck`、`test`、
  `lint`、`pack:smoke` を定義する。

理由:

- CLI の規模に対して最小構成であり、責務が見えやすい。
- ネイティブ依存候補を含むため、初期段階では bundler を避けた方が
  `npm pack` 後の実行検証を単純に保てる。
- `pack:smoke` を最初から scripts 化しておくと、
  [仕様書](../spec.md) の配布契約を継続的に確認しやすい。

### 3.6 引数解析方針

- v1 の引数解析は `process.argv.slice(2)` を直接処理する。
- 外部の引数解析ライブラリは導入しない。
- サポート対象コマンドが少なく、`run -- ...` の境界も明示的なため、
  手書き実装で十分と判断する。
- オプション体系が将来増えた場合にだけ再評価する。

## 4. `dotenvx` 依存解決

`dotenvx` は本 CLI の同梱依存として解決する。

公開パッケージ前提:

- `dotenvx` は公開パッケージの `dependencies` に置く。
- `devDependency`、`peerDependency`、
  グローバルインストール前提では扱わない。

探索順序:

1. 自パッケージ配下の実行ファイルを解決する。
2. npm が生成するバイナリ shim を優先して使う。
3. 直接パス解決ができない場合だけ依存関係エラーとして失敗する。

決定事項:

- ユーザー環境のグローバル `dotenvx` には依存しない。
- `PATH` 上の同名実行ファイル探索はしない。
- 解決失敗時は終了コード `4` を返す。
- エラーメッセージは再インストールまたは依存確認を促す。

## 5. 子プロセス起動

### 5.1 共通原則

- 親プロセスの環境は変更しない。
- 子プロセスにだけ `DOTENV_PRIVATE_KEY` を渡す。
- 可能な限りシェル非経由で起動する。

起動処理は `spawn` 系 API を前提にし、
標準入出力は既定で親へ透過する。

共通オプション:

- `cwd` は現在の作業ディレクトリをそのまま使う。
- `env` は `process.env` をコピーして `DOTENV_PRIVATE_KEY` を上書きする。
- `stdio` は `inherit` を既定とする。

### 5.2 Windows 例外

- `.cmd` または `.bat` 解決時のみシェル経由を許可する。

詳細方針:

- 先頭トークンが `/`、`\`、`./`、`../`、ドライブレター付きパスの
  いずれかに該当する場合は、まずそのパスを直接判定する。
- それ以外の場合は `PATH` を順に探索し、拡張子が明示されていなければ
  `PATHEXT` の候補を付与して解決する。
- 解決結果の拡張子が `.cmd` または `.bat` なら、
  そのときだけ `shell: true` を許可する。
- 解決結果が `.exe`、`.com`、拡張子なし実行可能ファイル、
  Node.js スクリプトなら非シェル起動を維持する。
- 解決に失敗した場合は、シェルへフォールバックせず、
  起動前失敗として終了コード `4` へ変換する。
- ユーザー指定の先頭コマンドが `.cmd` または `.bat` に解決される場合だけ
  `shell: true` 相当を許可する。
- 判定責務は共通ランタイム層に置き、`run` コマンド側へ漏らさない。
- `.exe`、`.com`、Node.js スクリプトは通常どおり非シェル起動する。

### 5.3 終了伝播

- 子プロセスが数値の終了コードを返したら、そのまま返す。
- シグナル終了なら親 CLI も同じシグナルで終了させる。
- 起動前失敗は CLI 独自エラーとして終了コード `4` を返す。

### 5.4 `dotenvx run` の組み立て

`run` コマンドが起動する子プロセス列は次の形に固定する。

```text
dotenvx run -- <command> [args...]
```

- `dotenvx` 自体の追加オプションは v1 では注入しない。
- ユーザー引数の再引用や 1 文字列連結は行わない。
- 受け取った配列をそのまま子プロセス引数に変換する。

## 6. 標準出力と標準エラー

- 成功メッセージは標準出力へ出す。
- 使用法エラーと実行時エラーは標準エラーへ出す。
- `run` の成功時は原則として本体メッセージを出さず、
  子プロセス出力だけを流す。
- 秘密値、鍵文字列、完全な環境ダンプは出力しない。
- 例外をログ化する場合も `DOTENV_PRIVATE_KEY` を必ず除外する。

### 6.1 メッセージ構成

- 1 行目に失敗原因を短く示す。
- 2 行目以降で必要なら再実行方法を示す。
- 内部スタックトレースは既定で出さない。

## 7. 実装メモ

- コマンド個別ロジックと共通ロジックを分離する。
- テストしやすさのため、プロセス起動は薄いラッパー経由にする。
- `resolveDotenvxBinary`、`spawnChildProcess`、`formatCliError` のように
  役割を薄く分けると単体試験を書きやすい。

## 8. 前提

- v1 の出力形式はプレーンテキストのみとする。
- JSON 出力や静音モードは扱わない。
- 対話的な確認プロンプトは行わない。
