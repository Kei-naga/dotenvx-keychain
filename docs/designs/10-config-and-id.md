# 設計: 設定ファイルと ID

## 1. 目的

- [仕様書](../spec.md) の 5.2、6、7.1 を詳細化する。
- 設定ファイルの読み書き契約と ID 解決規則を実装可能な粒度で固める。

## 2. 設計対象

- `.dotenvx-keychain` の JSON 形式
- 設定ファイルの読込、検証、書込
- 明示 ID のバリデーション
- 自動生成 ID の正規化手順
- `run` 時のプロジェクトルート探索

## 3. 入出力の契約

### 3.1 入力

- カレントディレクトリの絶対パス
- `init` に与えられた明示 ID 文字列
- 探索対象ディレクトリに存在する `.dotenvx-keychain`
- `run` 実行時の親ディレクトリ列

### 3.2 出力

- 正規化済みの ID
- プロジェクトルートの絶対パス
- 設定ファイルの読込結果
- 使用法エラーまたは構文エラー

### 3.3 呼び出し境界

- `resolveInitId(cwd, explicitId)`:
  `init` が使う ID と出力先パスを決める。
- `resolveRunProject(cwd)`:
  `run` が使うプロジェクトルートと ID 解決元を決める。
- `readConfig(path)`:
  設定ファイルを検証付きで読み込む。
- `writeConfig(path, id)`:
  設定ファイルを既定フォーマットで書き込む。

## 4. 詳細設計

### 4.1 設定ファイルのスキーマ

- ファイル名は `.dotenvx-keychain` に固定する。
- エンコーディングは UTF-8 とし、BOM は許容する。
- JSON ルートは object でなければならない。
- 必須キーは `id` のみとする。
- `id` は非空文字列でなければならない。
- 未知の追加キーは読み込み時に無視する。
- 未知キーを再保存時に保持する必要はない。
- 機密値らしきキー名が含まれていても特別扱いせず、
  v1 では読み捨てる。

```json
{
  "id": "my-app-v2"
}
```

- 書き込み時の整形は 2 スペースインデントと末尾改行 1 つに統一する。

### 4.2 読み込み処理

読込処理は次の順で進める。

1. 対象パスが存在しなければ `not-found` を返す。
2. ファイルとして読めなければ I/O エラーを返す。
3. UTF-8 文字列として読み込む。
4. JSON パースに失敗したら構文エラーを返す。
5. ルートが object でなければ構文エラーを返す。
6. `id` が文字列でなければ構文エラーを返す。
7. `id` に対して明示 ID と同じバリデーションを行う。
8. 成功時は `id` と設定ファイルパスを返す。

戻り値は例外をそのまま漏らさず、少なくとも次の分類を持つ。

- `not-found`
- `invalid-json`
- `invalid-schema`
- `io-error`

### 4.3 書き込み処理

- 出力先は対象ルート直下の `.dotenvx-keychain` とする。
- 書き込む JSON は `{"id": string}` のみとする。
- 既存ファイルがあれば全面上書きする。
- 追加キーを保持するマージ書き込みは行わない。
- 改行コードは `\n` に統一する。
- v1 では一時ファイルを使わず直接上書きでよい。
- 書き込み失敗時は `init` 側でロールバック判断を行う。

出力例:

```json
{
  "id": "my-app-v2"
}
```

### 4.4 明示 ID バリデーション

- 許容長は 1 文字以上 128 文字以下とする。
- 許容文字は `a-z`、`A-Z`、`0-9`、`.`、`_`、`-` とする。
- 先頭末尾を含め空白は許可しない。
- `/` と `\` は禁止する。
- Unicode 正規化は行わない。入力文字列をそのまま評価する。
- バリデーションの判定は次の正規表現相当とする。

```text
^[A-Za-z0-9._-]{1,128}$
```

- 不正時は終了コード `2` を返す入力エラーに変換する。
- エラーメッセージには許容文字種を明示する。
- 入力値そのものは表示してよいが、前後空白は引用して示す。

### 4.5 自動生成 ID

自動生成 ID は「同じ実ディレクトリなら安定し、別ディレクトリなら
高確率で衝突しない」ことを目的とする。

手順は次のとおり。

1. 対象ディレクトリの実パスを取得する。
2. Windows ではドライブレターだけを小文字化する。
3. すべての区切りを `/` に正規化する。
4. 末尾区切りを除いた最後の要素を `basename` とする。
5. 正規化後パスの SHA-256 を 16 進小文字で求める。
6. ハッシュ先頭 12 文字を `hash12` とする。
7. `basename-hash12` を最終 ID とする。

補足:

- `basename` 側は追加の文字種変換を行わない。
- ただし実運用で不正文字を含む可能性があるため、
  自動生成後にも最終 ID バリデーションを通す。
- 自動生成結果がバリデーションを通らない場合は、
  `basename` を固定文字列 `project` に置き換えて再生成する。
- それでも失敗する場合は実装不整合として終了コード `4` を返す。

例:

- `C:\Users\keita\source\my-app` -> `my-app-1a2b3c4d5e6f`
- `/home/keita/work/my-app` -> `my-app-7f8e9d0c1b2a`

### 4.6 `run` 時のルート探索

探索アルゴリズムは次のとおり。

1. `cwd` を探索開始地点とする。
2. 現在ディレクトリ直下に `.dotenvx-keychain` があるか確認する。
3. あればそのディレクトリをプロジェクトルートとして採用する。
4. なければ親ディレクトリへ 1 段上がる。
5. ルートディレクトリに到達するまで 2 から 4 を繰り返す。
6. 1 つも見つからなければ、元の `cwd` をプロジェクトルートとする。

決定事項:

- 複数候補がある場合は最初に見つかった最も近いものを使う。
- 設定ファイルが見つからないときだけ自動 ID へフォールバックする。
- `run` は探索結果のパスをそのまま `resolveRunProject` の結果として返す。

### 4.7 `init` と `run` の解決差分

- `init` は常に `cwd` を対象ルートとし、親探索をしない。
- `run` だけが親探索を行う。
- `init` の既存設定再利用は `cwd` 直下だけを見る。
- `run` の設定再利用は最寄り祖先の設定を使う。

## 5. 実装メモ

- この文書で決めた処理は `config-store` と `id-resolver` のような
  小さなモジュールへ分ける。
- 設定ファイル処理は CLI コマンド本体から直接触らず、
  専用モジュール経由とする。
- `readConfig` の戻り値で `not-found` と `invalid-*` を分けると、
  `run` のエラー分岐を単純化できる。

## 6. 前提

- v1 では 1 設定ファイルにつき 1 ID のみ扱う。
- 機密値は設定ファイルへ保存しない。
- `.dotenvx-keychain` は Git 管理対象に含まれてよい。
