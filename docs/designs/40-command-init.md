# 設計: `init` コマンド

## 1. 目的

- [仕様書](../spec.md) の 8.1 を詳細化する。
- `init` の正常系、後処理、ロールバックを実装順に落とし込む。

## 2. 設計対象

- ID 解決順序
- 鍵供給元の解決と `dotenvx` との連携
- シークレット保存
- `.dotenvx-keychain` 書き込み
- `.env.keys` 削除
- 失敗時ロールバック

対象外:

- `run` の親探索ロジック
- `list` と `remove` の列挙契約

## 3. シーケンス

1. カレントディレクトリを対象ルートとして確定する。
2. 明示 ID、既存設定、自動生成の順で使用 ID を決める。
3. 同一 ID の既存鍵がストアにあるか確認する。
4. なければ親プロセスの `DOTENV_PRIVATE_KEY` を確認する。
5. それにも値がなければ、既存の `.env.keys` から
  `DOTENV_PRIVATE_KEY` を取得できるか確認する。
6. 既存設定再利用で 3 から 5 のどれにも値がなければ、
  既存鍵不足として失敗する。
7. 新規セットアップで 3 から 5 のどれにも値がなければ、
  新規鍵不足として失敗する。
8. 得られた鍵を必要に応じてシークレットストアへ保存する。
9. `.dotenvx-keychain` を書き込む。
10. `.env.keys` があれば削除する。
11. 成功メッセージを出す。

正常系では、設定ファイル更新完了後にだけ成功扱いへ進む。

## 4. ステップ別詳細

### 4.1 ID 解決

ID 解決順序は仕様どおり次の 3 段階に固定する。

1. 明示引数 `<id>` があればそれを使う。
2. 明示引数がなく、`cwd/.dotenvx-keychain` があれば再利用する。
3. どちらもなければ `cwd` から自動生成する。

決定事項:

- `init` は親ディレクトリ探索を行わない。
- 明示 ID と既存設定 ID が異なっていても明示 ID を優先する。
- 明示 ID は入力エラーとして先に検証する。

### 4.2 鍵供給元解決と `dotenvx` 実行

`init` は次の優先順位で鍵供給元を決める。

1. `secretStore.get(id)` で取得できる既存ストア値
2. 親プロセスの `DOTENV_PRIVATE_KEY`
3. `dotenvxAdapter.readPrivateKey(projectRoot)` で読める既存ローカル鍵

既存設定再利用で決まった ID に対して、1 から 3 のどれにも
値がない場合は、既存鍵不足として失敗する。

新規セットアップで 1 から 3 のどれにも値がない場合は、
新規鍵不足として失敗する。

`dotenvx` 呼び出しが必要な場合だけ、
[35-dotenvx-adapter.md](./35-dotenvx-adapter.md) で定義した
薄い抽象を経由する。

この抽象の責務:

- 対象プロジェクトに対して、既存のローカル鍵ソースから
  `DOTENV_PRIVATE_KEY` を取得する。
- 呼び出し後に `.env.keys` が残る可能性を許容する。
- 秘密鍵文字列だけを `init` コマンドへ返す。
- 外部コマンドの標準出力、標準エラーから秘密値を漏らさない。

`init` 側の成功判定:

- 鍵文字列が非空文字列として確定すること。
- 依存解決とプロセス終了が成功であること。

### 4.3 ストア保存

- 保存は `secretStore.set(id, privateKey)` で行う。
- 同一 ID の既存エントリがあっても上書きを許可する。
- 既存ストア値をそのまま再利用する場合は、
  `secretStore.set` を再実行しなくてよい。
- 親プロセスの `DOTENV_PRIVATE_KEY` を採用した場合は、
  その値を `secretStore.set` で保存してから次へ進む。
- 保存成功後にだけ設定ファイル書き込みへ進む。
- 保存時点では旧 ID の削除を試みない。

### 4.4 設定ファイル更新

- `writeConfig(projectRoot/.dotenvx-keychain, id)` を呼ぶ。
- 書き込む内容は `{ "id": "..." }` のみとする。
- 既存ファイルがあれば全面上書きする。
- 旧 ID が異なっていても、旧 ID のキーチェーン項目は残す。

### 4.5 `.env.keys` 後処理

後処理は次の順で行う。

1. `projectRoot/.env.keys` の存在を確認する。
2. 存在しなければ何もしない。
3. 存在すれば削除を 1 回試みる。
4. 削除成功なら成功扱いを継続する。
5. 削除失敗なら終了コード `5` で失敗する。

削除失敗時の扱い:

- エラーメッセージには残存パスを示す。
- 鍵の内容は出さない。
- この失敗はストア保存成功より優先して最終結果を失敗にする。

### 4.6 成功条件

`init` が成功とみなされるのは、次をすべて満たした場合だけとする。

- 使用 ID が確定している。
- 秘密鍵が取得できている。
- ストア保存が成功している。
- 設定ファイル書き込みが成功している。
- `.env.keys` が最終的に存在しない。

## 5. 異常系

- 明示 ID 不正:
  終了コード `2`。
- 既存鍵不足:
  終了コード `3`。
- 新規鍵不足:
  終了コード `3`。
- `.env.keys` からの鍵取得失敗:
  `.env.keys` 削除を試みたうえで終了コード `4`。
- ストア保存失敗:
  `.env.keys` 削除を試みたうえで終了コード `4`。
- 設定ファイル書き込み失敗:
  ストアロールバックを試み、終了コード `4`。
- `.env.keys` 削除失敗:
  終了コード `5`。
- 未対応プラットフォーム:
  `.env.keys` 削除を試みたうえで終了コード `4`。
- ネイティブストア利用不可:
  `.env.keys` 削除を試みたうえで終了コード `4`。

## 6. ロールバック方針

ロールバック対象は「今回の `init` 実行で新たに保存または上書きした
ストアエントリ」に限る。

実施条件:

- `secretStore.set` 成功後に `writeConfig` が失敗した場合だけ、
  `secretStore.remove(id)` を試みる。
- `dotenvx` 実行失敗や保存前失敗ではストアロールバックは不要。

ロールバック失敗時の扱い:

- 最終終了コードは `4` のままとする。
- メッセージに「ストアの手動確認が必要」と付記する。
- それでも秘密鍵の内容は出力しない。

## 7. 成功メッセージ案

- 標準出力に 2 行程度の短いメッセージを出す。
- 表示する情報は使用 ID と設定ファイル配置先だけとする。
- 表示しない情報は秘密鍵文字列、ハッシュ全文、
  バックエンドの内部識別子とする。

例:

```text
Initialized dotenvx-keychain with id: my-app-v2
Wrote config: .dotenvx-keychain
```

## 8. 前提

- 旧 ID のキーチェーンエントリは自動削除しない。
- `.env.keys` 残存はセキュリティ上の失敗とみなす。
- v1 はチーム向けの鍵配布自動化を持たない。
- `dotenvx` の具体サブコマンドは adapter の内側に閉じ込める。
- `init` は v1 で新規鍵生成や `.env` の暗号化を行わない。
