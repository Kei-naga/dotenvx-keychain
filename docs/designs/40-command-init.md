# 設計: `init` コマンド

## 1. 目的

- [仕様書](../spec.md) の 8.1 を詳細化する。
- `init` の既存鍵取り込み経路と、新規ブートストラップ経路を
  実装順に落とし込む。

## 2. 設計対象

- ID 解決順序
- 鍵供給元の解決と `dotenvx` との連携
- シークレット保存
- bootstrap 成果物としての `.env` 反映
- `.dotenvx-keychain` 書き込み
- `.env.keys` 削除
- 失敗時ロールバック

対象外:

- `run` の親探索ロジック
- `list` と `remove` の列挙契約

## 3. シーケンス

1. カレントディレクトリを対象ルートとして確定する。
2. 明示 ID、既存設定、自動生成の順で使用 ID を決める。
3. `cwd/.dotenvx-keychain` が実在するかを確認する。
4. 同一 ID の既存鍵がストアにあるか確認する。
5. なければ親プロセスの `DOTENV_PRIVATE_KEY` を確認する。
6. それにも値がなければ、既存の `.env.keys` / 暗号化済み `.env` から
   `DOTENV_PRIVATE_KEY` を取得できるか確認する。
7. 4 から 6 のどれにも値がなく、かつ既存設定ファイルがあるなら、
   既存鍵不足として失敗する。
8. 4 から 6 のどれにも値がなく、かつ `.env` が既に暗号化済みなら、
   既存暗号化状態に対応する鍵不足として失敗する。
9. 4 から 8 のどれにも該当しない新規プロジェクトなら、
   隔離した一時ディレクトリで bootstrap を行う。
10. 得られた鍵を必要に応じてシークレットストアへ保存する。
11. bootstrap 成果物がある場合はプロジェクトルートの `.env` を更新する。
12. `.dotenvx-keychain` を書き込む。
13. `.env.keys` があれば削除する。
14. 成功メッセージを出す。

正常系では、設定ファイル更新と `.env.keys` 後処理まで完了した後にだけ成功扱いへ進む。

## 4. ステップ別詳細

### 4.1 ID 解決

ID 解決順序は仕様どおり次の 3 段階に固定する。

1. 明示引数 `<id>` があればそれを使う。
2. 明示引数がなく、`cwd/.dotenvx-keychain` があれば再利用する。
3. どちらもなければ `cwd` から自動生成する。

決定事項:

- `init` は親ディレクトリ探索を行わない。
- 明示 ID と既存設定 ID が異なっていても明示 ID を優先する。
- 明示 ID は入力エラーとして先に検証する。
- bootstrap 可否判定は、解決した ID の由来だけでなく
  `cwd/.dotenvx-keychain` の実在と `.env` の状態も見る。

### 4.2 鍵供給元解決と `dotenvx` 実行

`init` は次の優先順位で鍵供給元を決める。

1. `secretStore.get(id)` で取得できる既存ストア値
2. 親プロセスの `DOTENV_PRIVATE_KEY`
3. `dotenvxAdapter.readPrivateKey(projectRoot)` で読める既存ローカル鍵
4. `dotenvxAdapter.bootstrapProjectEnv(projectRoot)` による新規ブートストラップ

制約:

- 3 は `.env.keys` が残っている既存ローカル状態の救済として使う。
- `cwd/.dotenvx-keychain` が存在する場合は 4 を使わない。
- `.env` が既に暗号化済みで 1 から 3 のどれにも鍵がない場合も、
  4 を使わない。
- `dotenvx` 呼び出しの具体コマンド列は
  [35-dotenvx-adapter.md](./35-dotenvx-adapter.md) の内側へ閉じ込める。

### 4.3 ストア保存

- 保存は `secretStore.set(id, privateKey)` で行う。
- 同一 ID の既存エントリがあっても上書きを許可する。
- 既存ストア値をそのまま再利用する場合は、
  `secretStore.set` を再実行しなくてよい。
- 親プロセスの `DOTENV_PRIVATE_KEY`、ローカル鍵、bootstrap 鍵を採用した場合は、
  `secretStore.set` で保存してから次へ進む。

### 4.4 `.env` 反映

- `.env` を更新するのは bootstrap 経路だけとする。
- bootstrap 前の `.env` が存在すれば、その内容を退避してから上書きする。
- bootstrap 前の `.env` が存在しなければ、新しい `.env` を作成する。
- 空プロジェクトの bootstrap 後は、`DOTENV_PUBLIC_KEY` と説明コメントだけを
  含む `.env` になることがある。

### 4.5 設定ファイル更新

- `writeConfig(projectRoot, id)` を呼ぶ。
- 書き込む内容は `{ "id": "..." }` のみとする。
- 既存ファイルがあれば全面上書きする。
- 旧 ID が異なっていても、旧 ID のキーチェーンエントリは残す。

### 4.6 `.env.keys` 後処理

後処理は次の順で行う。

1. `projectRoot/.env.keys` の存在を確認する。
2. 存在しなければ何もしない。
3. 存在すれば削除を 1 回試みる。
4. 削除成功なら成功扱いを継続する。
5. 削除失敗なら終了コード `5` で失敗する。

削除失敗時の扱い:

- エラーメッセージには残存パスを示す。
- 鍵の内容は出さない。
- この失敗はストア保存成功より優先して最終結果を失敗にする。

### 4.7 成功条件

`init` が成功とみなされるのは、次をすべて満たした場合だけとする。

- 使用 ID が確定している。
- 秘密鍵が取得できている。
- ストア保存が成功している。
- bootstrap 経路なら `.env` 反映が成功している。
- 設定ファイル書き込みが成功している。
- `.env.keys` が最終的に存在しない。

## 5. 異常系

- 明示 ID 不正:
  終了コード `2`。
- 既存設定ファイルがある状態での鍵不足:
  終了コード `3`。
- 既存の暗号化済み `.env` に対応する鍵不足:
  終了コード `3`。
- `.env.keys` / 暗号化済み `.env` からの鍵取得失敗:
  終了コード `4`。
- bootstrap 失敗:
  終了コード `4`。
- ストア保存失敗:
  終了コード `4`。
- `.env` 更新失敗:
  ストアロールバックと `.env` 復元を試みたうえで終了コード `4`。
- 設定ファイル書き込み失敗:
  ストアロールバックと必要なら `.env` 復元を試み、終了コード `4`。
- `.env.keys` 削除失敗:
  終了コード `5`。
- 未対応プラットフォーム:
  終了コード `4`。
- ネイティブストア利用不可:
  終了コード `4`。

## 6. ロールバック方針

ロールバック対象は次の 2 つに限る。

- 今回の `init` 実行で新たに保存または上書きしたストアエントリ
- bootstrap 経路で今回書き換えた `.env`

実施条件:

- `secretStore.set` 成功後に `.env` 更新が失敗した場合は、
  `secretStore.remove(id)` と `.env` 復元を試みる。
- `secretStore.set` 成功後に `writeConfig` が失敗した場合は、
  `secretStore.remove(id)` と必要なら `.env` 復元を試みる。
- bootstrap 前失敗では `.env` 復元もストアロールバックも不要。

`.env` 復元ルール:

- bootstrap 前に `.env` が存在したなら元の内容へ戻す。
- bootstrap 前に `.env` が存在しなかったなら削除する。

ロールバック失敗時の扱い:

- 最終終了コードは `4` のままとする。
- メッセージに「手動確認が必要」と付記する。
- それでも秘密鍵の内容は出力しない。

## 7. 成功メッセージ案

- 標準出力に 2 行程度の短いメッセージを出す。
- 表示する情報は使用 ID と設定ファイル配置先だけとする。
- 表示しない情報は秘密鍵文字列、ハッシュ全文、
  バックエンドの内部識別子とする。

例:

```text
Initialized dotenvx-keychain with id: my-app-v2
Wrote config: .dotenvx-keychain
```

## 8. 前提

- 旧 ID のキーチェーンエントリは自動削除しない。
- `.env.keys` 残存はセキュリティ上の失敗とみなす。
- v1 はチーム向けの鍵配布自動化を持たない。
- `dotenvx` の具体サブコマンドは adapter の内側に閉じ込める。
- 新規ブートストラップは既存暗号化状態の鍵ローテーション手段ではない。
