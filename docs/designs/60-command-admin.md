# 設計: `list` / `remove` コマンド

## 1. 目的

- [仕様書](../spec.md) の 8.3、8.4 を詳細化する。
- 管理系コマンドの出力契約と削除契約を明確にする。

## 2. 設計対象

- `list` の列挙と整列
- `remove` の完全一致削除
- 成功時、未検出時、障害時の出力

対象外:

- 設定ファイルの自動更新
- プロジェクトルート探索

## 3. `list` 設計

### 3.1 シーケンス

1. バックエンドを初期化する。
2. 名前空間内の ID を列挙する。
3. 昇順ソートする。
4. 1 行 1 ID で標準出力へ出す。

このコマンドはカレントディレクトリや設定ファイルに依存しない。

### 3.2 出力契約

- 空件数時は標準出力を空のまま終了コード `0` で終える。
- 出力は 1 行 1 ID のプレーンテキストのみとする。
- 並び順は Unicode コードポイント順の昇順でよい。
- 鍵文字列、伏字、保存日時、プラットフォーム情報は出さない。
- 重複 ID がバックエンドから返ってきた場合は重複除去してから出す。

例:

```text
app-a
my-app-v2
sample-project
```

## 4. `remove` 設計

### 4.1 シーケンス

1. 入力 ID を検証する。
2. バックエンドを初期化する。
3. 完全一致で削除を試みる。
4. 結果メッセージを返す。

`remove` は設定ファイルを一切変更しないため、
ID が現在のプロジェクトで使われているかどうかは確認しない。

### 4.2 削除契約

- 指定 ID に完全一致するエントリだけを削除する。
- 設定ファイルを変更しない。
- `.env` と `.env.keys` を変更しない。
- 対象未検出時は終了コード `3` を返す。
- 削除成功時は終了コード `0` を返す。

### 4.3 入力検証

- `remove` の ID は `init` の明示 ID と同じ規則で検証する。
- 不正文字や空文字は終了コード `2` とする。

## 5. 異常系

- 未対応プラットフォーム:
  終了コード `4`。
- バックエンド利用不可:
  終了コード `4`。
- 削除対象未検出:
  終了コード `3`。
- `list` の列挙失敗:
  終了コード `4`。
- `remove` の実行障害:
  終了コード `4`。

## 6. メッセージ方針

- `list` 成功時は ID 一覧以外を出さない。
- `remove` 成功時は削除した ID を 1 行で示す。
- 未検出時は対象 ID が存在しないことを示す。
- 障害時はバックエンド利用不可または削除失敗を短く伝える。

例:

```text
Removed key: my-app-v2
```

## 7. 前提

- `list` は人間向けプレーンテキスト出力のみとする。
- `remove` は対話確認を行わない。
- `remove` は成功時も設定ファイルの不整合チェックをしない。
