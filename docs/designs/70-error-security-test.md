# 設計: エラー、セキュリティ、テスト観点

## 1. 目的

- [仕様書](../spec.md) の 9、10、11 を横断要件として詳細化する。
- コマンド個別文書へ分散しがちな失敗分類、秘密値保護、検証観点を集約する。

## 2. 設計対象

- 終了コード
- エラー分類
- メッセージの秘密値マスキング
- 受け入れ基準から導くテスト観点

対象外:

- 各 OS の実装詳細
- npm パッケージの公開パイプライン

## 3. 終了コード表

| コード | 意味 | 主な発生箇所 |
| --- | --- | --- |
| `0` | 成功 | 全コマンド |
| `2` | 使用法または入力値エラー | 引数不足、ID 不正 |
| `3` | 指定 ID の鍵が見つからない | `run`、`remove` |
| `4` | `dotenvx` または OS ストア連携の失敗 | `init`、`run`、`list`、`remove` |
| `5` | セキュリティ上の後処理失敗 | `init` の `.env.keys` 削除失敗 |

運用方針:

- 同一コマンド内で複数失敗が起きた場合は、
  より重大な後処理失敗を優先して返す。
- 子プロセスが返した終了コードは `run` の成功経路ではそのまま返す。

## 4. エラー分類

### 4.1 入力エラー

- 想定ケースは未知コマンド、`run` の子コマンド不足、
  `remove` の ID 不足、明示 ID の文字種不正とする。
- 表示文面は「理由 + 使用法」の順で短く出す。
- 再試行方法は 1 つだけ提示し、複数案内を並べない。

### 4.2 実行時エラー

- `dotenvx` 不在
- ストア利用不可
- 設定ファイル破損
- 鍵未検出

分類ごとの原則:

- `dotenvx` 不在:
  依存関係エラーとして終了コード `4`。
- ストア利用不可:
  環境要件エラーとして終了コード `4`。
- 設定ファイル破損:
  構文またはスキーマ不正として終了コード `4`。
- 鍵未検出:
  ユーザー再設定可能エラーとして終了コード `3`。

### 4.3 後処理エラー

- `.env.keys` 削除失敗
- ロールバック失敗

後処理エラーの原則:

- `.env.keys` 削除失敗は常に終了コード `5` を優先する。
- ロールバック失敗は主エラーを維持しつつ補足文面で通知する。

### 4.4 メッセージ設計原則

- 秘密値を含む文字列連結を禁止する。
- パス、ID、コマンド名だけを可視情報として扱う。
- 内部例外の `message` をそのまま表示しない。

## 5. セキュリティ要件

- 鍵文字列を標準出力、標準エラー、ログに出さない。
- 例外やスタックトレースへ秘密値を混入させない。
- `run` の環境変数注入を子プロセス系統に限定する。
- ワーキングツリーに鍵ファイルを残さない。

追加要件:

- `list` は存在確認に必要な ID 以外を出さない。
- `remove` は削除対象の ID 以外を出さない。
- `init` の失敗時も `.env.keys` 削除を優先して試みる。
- デバッグログを将来追加する場合も既定では無効にする。

## 6. テスト観点

### 6.1 単体テスト

- ID バリデーション
- 自動生成 ID の安定性
- ルート探索
- エラー変換

追加観点:

- 設定ファイル読込の `not-found` と `invalid-json` の分離
- Windows シェル経由判定
- メッセージ整形時の秘密値除外

### 6.2 結合テスト

- `init` 正常系
- `run` 正常系
- `list` の昇順出力
- `remove` の完全一致削除

追加観点:

- `init` で既存設定 ID 再利用
- `run` で設定ファイル未発見時の自動 ID フォールバック
- `init` 失敗時の `.env.keys` 削除
- `remove` が設定ファイルを変更しないこと
- `npm pack` した tarball を一時導入した状態で、
  `dotenvx-keychain` と `dxk` が同じ CLI へ到達すること
- 上記環境でグローバル `dotenvx` がなくても、
  同梱依存の解決に成功すること

### 6.3 プラットフォーム観点

- macOS
- Windows
- Linux
- 未対応 OS

観点ごとの最低確認項目:

- macOS:
  保存、取得、列挙、削除が同じ ID 契約で動く。
- Windows:
  `.cmd` または `.bat` 実行時の起動条件が保たれる。
- Linux:
  Secret Service 利用不可時に平文へ落ちない。
- 未対応 OS:
  全コマンドが明示的に終了コード `4` で失敗する。

### 6.4 受け入れ基準との対応

- [仕様書](../spec.md) 11 章の各項目に対して、
  少なくとも 1 つの結合テストを割り当てる。
- OS 依存で自動化しにくい項目は手動確認項目として残す。

### 6.5 自動化境界

- 単体テストは全 OS で同一内容を自動実行する。
- コマンド結合テストは Secret Store をモック実装へ差し替えて自動化する。
- 実ストアを使うスモークテストは、
  `set/get/list/remove` の最小契約確認だけに絞って OS 別に分ける。
- `npm pack` 後の tarball を一時ディレクトリへ導入し、
  `dotenvx-keychain` と `dxk` の両方が起動することを
  リリース前の必須スモークとする。
- Linux の Secret Service 利用不可ケースは、
  自動化環境で安定再現できない場合に限り手動確認へ残してよい。

## 7. 前提

- v1 では人間向けメッセージのみを保証する。
- セキュリティ要件は利便性より優先する。
- 例外情報よりも秘密値非露出を優先する。
