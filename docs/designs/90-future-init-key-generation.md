# 将来検討: `init` の自動鍵生成

## 1. 目的

- v1 では採用しない `init` の自動鍵生成案を、
  現行仕様と混ぜずに保管する。
- 将来、`dotenvx-keychain` 自体が新規鍵生成まで担う場合の
  制約と検討事項を残す。

## 2. 現時点の位置づけ

- v1 は A 案を採用し、`init` を「既存鍵の取り込みと隔離」に限定する。
- この文書が扱う B 案は、将来検討であり、
  現在の [仕様書](../spec.md) の契約には含めない。

## 3. B 案の概要

- `init` が `dotenvx` を呼び出して、
  新しい `DOTENV_PRIVATE_KEY` の生成まで担う。
- 生成した鍵を OS ストアへ保存し、
  `.env.keys` を成功時に削除する。
- 開発者は `dotenvx encrypt` を事前に実行しなくても、
  `init` だけで初期セットアップを開始できるようにする。

## 4. この案を v1 から外した理由

- 実コマンド確認では、`dotenvx encrypt` を `.env` 不在または空ファイルの
  ディレクトリで実行すると、新しい `.env` と `.env.keys` を生成した。
- その新しい `.env` には、親プロセス環境由来の変数群が
  暗号化されて書き込まれる挙動が見られた。
- そのため、`dotenvx encrypt` をプロジェクトルートで直接呼ぶ設計は、
  鍵生成のつもりで `.env` を意図せず変更する危険がある。

## 5. 将来採用する場合の必須条件

### 5.1 安全なブートストラップ方式

- `.env` 不在または空ファイルのプロジェクトでも、
  本来のプロジェクトファイルを汚さずに鍵だけ生成できること。
- そのために、隔離した一時ディレクトリで `dotenvx` を実行するか、
  それと同等に安全な方式を定義すること。
- 実行時の環境変数は allowlist ベースでサニタイズし、
  親プロセス環境の機密値を取り込まないこと。

### 5.2 仕様として追加が必要な事項

- どの `dotenvx` コマンド列を使うか
- `.env` を変更してよい条件と、変更してはいけない条件
- 失敗時に `.env`、`.env.keys`、OS ストアをどこまで巻き戻すか
- ロールバック不能時の最終優先順位をどうするか

### 5.3 テストとして追加が必要な事項

- `.env` 不在のディレクトリで、親環境変数を取り込まずに鍵だけ生成できること
- 空の `.env` で同じく安全に鍵だけ生成できること
- `.env` が既に暗号化済みのときに、意図しない鍵ローテーションを起こさないこと
- 失敗時に `.env.keys` や OS ストアの最終状態が仕様どおりであること

## 6. 再評価の入り口

次のいずれかが満たされたときに、B 案を再評価してよい。

1. `dotenvx` 側に鍵生成専用で安全な公式インターフェースが追加されたとき
2. 一時ディレクトリ + 環境変数サニタイズで十分安全な実装スパイクが通ったとき
3. A 案では導入負荷が高すぎるという明確なプロダクト判断が出たとき
